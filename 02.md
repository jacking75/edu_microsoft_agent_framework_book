## 2.1 AI Agent란 무엇인가?

### Agent의 정의와 역할
AI 에이전트는 단순히 질문에 답변하는 챗봇을 넘어서는 개념이다. 에이전트는 다음과 같은 특징을 가진다.

**자율성(Autonomy):** 에이전트는 주어진 목표를 달성하기 위해 스스로 어떤 단계를 거쳐야 할지 결정한다. 프로그래머가 모든 단계를 명시적으로 코딩할 필요가 없다.

**도구 사용(Tool Use):** 에이전트는 외부 함수나 API를 호출하여 실제 작업을 수행할 수 있다. 날씨를 확인하거나, 데이터베이스를 조회하거나, 파일을 생성하는 등의 작업이 가능하다.

**추론(Reasoning):** LLM을 사용하여 복잡한 문제를 분석하고 해결 방법을 찾는다. 단순한 패턴 매칭이 아니라 실제로 "생각"하는 것처럼 동작한다.

**반복 시도(Iteration):** 첫 시도가 실패하면 다른 방법을 시도한다. 에러를 분석하고 접근 방식을 수정하는 능력을 가진다.

**대화 기반(Conversational):** 사용자와 자연스러운 대화를 통해 요구사항을 파악하고 결과를 전달한다.

### 전통적인 프로그램과의 차이

**전통적인 프로그램:**

```csharp
// 명시적으로 모든 단계를 정의한다
string GetWeatherReport(string city)
{
    var weather = WeatherAPI.GetWeather(city);
    return $"{city}의 날씨는 {weather.Condition}이고 기온은 {weather.Temperature}도다.";
}
```

이 코드는 정확히 무엇을 할지 미리 정해져 있다. 예상치 못한 상황에 대응할 수 없다.

**AI 에이전트:**

```csharp
// 에이전트는 목표만 제시받는다
var agent = AIAgentBuilder
    .FromEnvironment()
    .Build("WeatherAgent",
        "사용자가 날씨를 물어보면 해당 도시의 날씨를 조회하여 알려준다.",
        new Delegate[] { GetWeather });

// 에이전트가 알아서 도구를 선택하고 호출한다
await agent.RunAsync("서울 날씨 어때?");
```

에이전트는 사용자의 질문을 분석하고, 날씨 조회가 필요하다고 판단하고, `GetWeather` 함수를 호출하고, 결과를 자연스러운 문장으로 만들어 답변한다.

### LLM 기반 의사결정 프로세스
AI 에이전트의 핵심은 Large Language Model(LLM)이다. LLM은 다음과 같은 방식으로 작동한다.

```
사용자 입력
    ↓
LLM 분석
    ↓
도구 호출 필요? ──아니오──→ 직접 답변 생성
    ↓ 예                        ↓
적절한 도구 선택             사용자에게 응답
    ↓
도구 실행
    ↓
결과 받기
    ↓
LLM 재분석
    ↓
사용자에게 응답
```

**단계별 설명:**

1. **입력 분석:** 사용자의 질문이나 요청을 이해한다. 의도를 파악하고 필요한 정보를 추출한다.
2. **계획 수립:** 목표를 달성하기 위해 어떤 단계가 필요한지 결정한다. 여러 도구를 순차적으로 사용해야 할 수도 있다.
3. **도구 선택:** 사용 가능한 도구 목록에서 가장 적합한 것을 선택한다. 함수의 이름, 설명, 파라미터를 분석하여 결정한다.
4. **실행 및 평가:** 도구를 실행하고 결과를 받는다. 결과가 만족스럽지 않으면 다른 접근 방식을 시도한다.
5. **응답 생성:** 모든 정보를 종합하여 사용자가 이해하기 쉬운 형태로 답변을 만든다.

### Tool Calling 개념
Tool Calling은 에이전트가 외부 함수를 호출하는 메커니즘이다.

**함수 등록:**

```csharp
string GetWeather(string location, string unit = "celsius")
{
    // 실제 날씨 조회 로직
    return $"{location}의 현재 기온은 20{unit}다.";
}

var agent = AIAgentBuilder
    .FromEnvironment()
    .Build("WeatherAgent", "날씨를 알려주는 에이전트다.",
        new Delegate[] { GetWeather });
```

**LLM의 해석:**

LLM은 이 함수를 다음과 같이 해석한다.

- **이름:** GetWeather
- **용도:** 특정 위치의 날씨 정보를 조회한다
- **필요한 정보:** location(필수), unit(선택, 기본값 celsius)

**호출 과정:**

사용자가 "서울 날씨 알려줘"라고 물으면 LLM이 "서울"을 추출 → `GetWeather("서울")` 호출 결정 → 함수 실행 → 결과를 자연스러운 답변으로 변환하는 흐름으로 진행된다.

> **중요:** 개발자는 함수만 제공하고, 언제 어떻게 호출할지는 LLM이 결정한다.

## 2.2 Microsoft Agent Framework 아키텍처

### Semantic Kernel과 AutoGen의 통합
Microsoft Agent Framework는 두 개의 기존 프로젝트를 통합하고 발전시킨 것이다.

`Agent Framework = Semantic Kernel (기반) + AutoGen (패턴) + Workflow (제어)`

**Semantic Kernel의 장점:** 엔터프라이즈급 안정성과 타입 안전성, Thread 기반 상태 관리, 풍부한 미들웨어 시스템, 다양한 LLM 공급자 지원.

**AutoGen의 장점:** 멀티 에이전트 오케스트레이션, 간단하고 직관적인 API, 이벤트 기반 아키텍처, 에이전트 간 통신 패턴.

### 주요 컴포넌트

```
┌──────────────────────────────────────────────┐
│           Application Layer                  │
│        (Your Agent Application)              │
└──────────────────┬───────────────────────────┘
                   │
┌──────────────────▼───────────────────────────┐
│         Agent Framework Core                 │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐   │
│  │  Agent   │  │ Workflow │  │  Thread  │   │
│  └──────────┘  └──────────┘  └──────────┘   │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐   │
│  │  Tools   │  │Middleware│  │ Context  │   │
│  └──────────┘  └──────────┘  └──────────┘   │
└──────────────────┬───────────────────────────┘
                   │
┌──────────────────▼───────────────────────────┐
│        Model Providers Layer                 │
│   OpenRouter │ Poe │ Any OpenAI-Compatible   │
└──────────────────────────────────────────────┘
```

**1. Agent (에이전트):** 개별 AI 에이전트다. LLM을 사용하여 사용자 입력을 처리하고, 도구를 호출하고, 응답을 생성한다.

```csharp
var agent = AIAgentBuilder
    .FromEnvironment()
    .Build("AssistantAgent", "당신은 도움을 주는 어시스턴트다.");
```

**2. Thread (스레드):** 대화의 상태를 관리한다. 이전 메시지들을 기억하고, 컨텍스트를 유지한다. 여러 턴에 걸친 대화를 가능하게 한다.

**3. Tools (도구):** 에이전트가 호출할 수 있는 함수들이다. 날씨 조회, 데이터베이스 쿼리, API 호출 등 실제 작업을 수행한다.

**4. Context Provider (컨텍스트 제공자):** 에이전트에게 추가 정보를 제공한다. 메모리, 지식 베이스, 사용자 프로필 등을 관리한다.

**5. Middleware (미들웨어):** 에이전트의 요청과 응답을 가로채서 처리한다. 로깅, 필터링, 인증 등의 횡단 관심사를 다룬다.

**6. Workflow (워크플로우):** 여러 에이전트와 함수를 연결한다. 복잡한 멀티 에이전트 시스템을 명시적으로 제어한다.

### 데이터 흐름 이해하기

```
사용자: "서울 날씨 알려줘"
    ↓
Agent → Thread: 메시지 저장
    ↓
Agent → LLM: 사용자 메시지 + 도구 정보 전송
    ↓
LLM → Agent: "GetWeather('서울') 호출 필요"
    ↓
Agent → Tool: GetWeather("서울") 실행
    ↓
Tool → Agent: "20도, 맑음"
    ↓
Agent → LLM: 도구 실행 결과 전송
    ↓
LLM → Agent: "서울은 현재 맑고 20도다."
    ↓
Agent → Thread: 응답 저장
    ↓
Agent → 사용자: "서울은 현재 맑고 20도다."
```

### Enterprise급 기능

**타입 안전성(Type Safety):**

```csharp
AIAgent agent = AIAgentBuilder.FromEnvironment().Build("Agent", "...");
RunResult result = await agent.RunAsync("질문");
string text = result.Text; // 컴파일 타임 타입 체크
```

**비동기 처리(Async/Await):** 모든 API가 비동기를 기본으로 한다. 높은 동시성과 효율적인 리소스 사용이 가능하다.

**의존성 주입(Dependency Injection):** ASP.NET Core의 DI 컨테이너와 자연스럽게 통합된다.

**미들웨어(Middleware):** OpenTelemetry를 지원하여 분산 추적, 메트릭, 로그를 수집한다.

### Workflow vs Agent 선택 기준

**단일 Agent를 사용할 때:** 작업이 비교적 단순하다(도구가 10개 미만), 실행 순서를 LLM이 결정해도 괜찮다, 대화형 인터페이스가 주 목적이다. 예시: 고객 지원 챗봇, 개인 비서.

**Workflow를 사용할 때:** 작업이 복잡하고 여러 단계가 필요하다, 실행 순서를 명시적으로 제어하고 싶다, 여러 에이전트의 협업이 필요하다, 장기 실행 프로세스다. 예시: 문서 처리 파이프라인, 복잡한 데이터 분석.

## 2.3 LLM 모델 비교 및 선택

### OpenRouter 지원 모델

이 가이드는 Azure 없이 **OpenRouter** 또는 **Poe**를 사용한다. OpenRouter는 다양한 모델을 단일 OpenAI 호환 API로 제공한다.

| 모델 ID | 설명 | 권장 용도 |
|---------|------|-----------|
| `anthropic/claude-sonnet-4-5` | Claude Sonnet - 균형잡힌 성능 | 대부분의 작업 (권장) |
| `anthropic/claude-haiku-4-5` | Claude Haiku - 빠르고 경제적 | 간단한 작업, 대량 처리 |
| `anthropic/claude-opus-4-5` | Claude Opus - 최고 성능 | 복잡한 분석, 고품질 요구 |
| `openai/gpt-4o` | GPT-4o | 범용 |
| `openai/gpt-4o-mini` | GPT-4o Mini | 경제적 범용 |
| `google/gemini-2.0-flash` | Gemini 2.0 Flash | 빠른 멀티모달 처리 |

### Poe 지원 모델

| 모델 ID | 설명 |
|---------|------|
| `claude-sonnet-4-20250514` | Claude Sonnet 4 |
| `claude-haiku-4-5` | Claude Haiku 4.5 |
| `gpt-4o` | GPT-4o |

### 모델 선택 기준

```
속도 우선 (대량 처리):   Haiku
균형 (대부분의 작업):   Sonnet ← 권장
품질 우선 (복잡한 분석): Opus
```

**AIAgentBuilder로 모델 선택:**

```csharp
// 환경 변수로 설정 (권장 - LLM_MODEL 환경 변수 변경으로 모델 전환)
var agent = AIAgentBuilder
    .FromEnvironment()
    .Build("MyAgent", "...");

// OpenRouter 직접 지정
var agent = AIAgentBuilder
    .WithOpenRouter(apiKey, "anthropic/claude-opus-4-5")
    .Build("PremiumAgent", "최고 품질의 분석이 필요한 에이전트다.");

// Poe 직접 지정
var agent = AIAgentBuilder
    .WithPoe(apiKey, "claude-haiku-4-5")
    .Build("FastAgent", "빠른 응답이 필요한 에이전트다.");
```

### Extended Thinking (추론) 기능
Claude의 독특한 기능 중 하나는 Extended Thinking이다. 모델이 답변하기 전에 "생각하는" 과정을 거치는 기능이다.

**장점:** 복잡한 문제에서 더 정확한 답변, 추론 과정의 투명성, 오류 가능성 감소.

**단점:** 더 많은 토큰 소비(비용 증가), 느린 응답 속도, 간단한 작업에는 불필요.

**적합한 사용 사례:** 복잡한 수학 문제, 다단계 추론 분석, 코드 디버깅, 전략적 의사결정.

### 토큰 비용 최적화

1. **적절한 모델 선택:** 모든 작업에 Opus를 쓸 필요는 없다.
2. **프롬프트 최적화:** 불필요한 정보를 제거하고 간결하게 작성한다.
3. **출력 길이 제한:** 불필요하게 긴 응답을 방지한다.
4. **배치 처리:** 가능하면 여러 요청을 모아서 처리한다.

한글은 영어보다 약 2~3배 많은 토큰을 소비한다는 점을 감안하여 비용을 예측해야 한다.

## 핵심 개념 정리

**AI Agent의 특징:** 자율적으로 도구를 선택하고 사용한다. LLM을 통해 복잡한 추론을 수행한다. 자연어로 사용자와 상호작용한다. 알고리즘을 미리 정의하기 어려운 작업에 적합하다.

**Agent Framework 구조:** Agent, Thread, Tools, Context, Middleware, Workflow로 구성된다. Semantic Kernel과 AutoGen의 장점을 통합했다. 엔터프라이즈급 안정성과 타입 안전성을 제공한다.

**LLM 선택:** Haiku는 빠르고 저렴하며 간단한 작업에, Sonnet은 균형잡힌 대부분의 작업에(권장), Opus는 최고 성능이 필요한 복잡한 작업에 사용한다.
